- ✅ **Connected** — verified wired end-to-end in code
- ⚠️ **Partial** — wired but with a caveat (missing recompute, missing pagination, dead-path duplication, etc.)
- ❌ **BROKEN / MISSING** — confirmed non-functional, schema mismatch, or absent

## 15.2 Dead / Duplicate Stack Cross-Reference

Two parallel implementations exist for the same business concept. QA/engineering must treat the **Live** column as ground truth and the **Dead** column as legacy code that should be deleted, not fixed, per `18-CTO-FINAL-REPORT.md`.

| Concept | Live path (used by Dashboard) | Dead/duplicate path (not reachable from Dashboard, still in routes) | Risk of leaving both |
|---|---|---|---|
| Party/customer ledger balance | `accountingController.getLedgerStatement` → `AccountingEntry` + `LedgerMaster` | `ledgerController.getPartyLedger`/`getAccountBalance` → `ledgerService` → `LedgerEntry` model, exposed at `GET /api/ledgers/:partyId` | Anyone (future dev, integration partner, mobile client) calling `/api/ledgers/:partyId` gets a **different, likely stale/incomplete** balance than the one shown in the app, with no error to indicate it's the wrong path |
| Purchase creation | `purchaseController.js` + `purchaseService.js` + `InventoryLot` | `purchase.controller.js` + `purchase.service.js` + `Inventory.js` model (note the near-identical filenames `purchase.controller.js` vs `purchaseController.js`) | High risk of a future contributor editing the wrong file and being confused why their change has no effect; `Inventory.js` model exists but is not populated by any live flow discovered in this audit |
| Dashboard page | `pages/Dashboard.jsx` (used in `App.jsx`, comment: *"Only the Legacy Dashboard is kept"*) | `pages/dashboard/Dashboard.jsx` (different file, different folder casing convention, not routed anywhere in `App.jsx`) | Orphaned code increases bundle analysis confusion and onboarding time; should be deleted or the routing comment clarified |
| Layout/navigation shell | None — `App.jsx` renders `Dashboard.jsx` directly with no layout wrapper for the tenant app | `components/layout/Sidebar.jsx`, `layouts/MainLayout.jsx` — both exist, reference each other and `AccountingPage.jsx`/`SettingsPage.jsx`, but are not mounted by any route in `App.jsx` | Same as above — confirms a partially-built "multi-page" navigation architecture was abandoned in favor of the modal-shell `Dashboard.jsx`, but the abandoned code was never removed |

## 15.3 Store-Layer Fan-Out (`frontend/src/store/useStore.js`)

Single ~1,570-line Zustand store is the de facto integration hub for nearly every screen in §15.1. Notable structural facts relevant to the integration matrix:

- Almost every modal in §15.1 destructures directly from `useStore()` without a selector (e.g. `Dashboard.jsx` line 111), meaning the *store*, not each individual screen's API call, is the actual dependency graph node between "Component" and "API" columns above.
- `bootstrapMasters()` and `refreshAllData()` (both called from `Dashboard.jsx`) appear to be the two umbrella actions that hydrate most of the store's arrays (`sales`, `purchases`, `parties`, `items`, `inventoryLots`, `jobWorkEntries`, `books`, etc.) in bulk on login/sync — meaning most individual modals do **not** independently fetch their own slice on open; they read whatever `bootstrapMasters`/`refreshAllData` already populated. This has two integration consequences:
  1. A modal opened immediately after a mutation elsewhere may show stale data until the next `refreshAllData()` call (no automatic per-mutation cache invalidation/re-fetch was confirmed for most flows outside Sales/Purchase's own store actions).
  2. The "Records Hub" and "Ledger Modal" screens' data completeness is entirely dependent on what this bulk hydration happened to load — reinforcing the ❌/⚠️ findings against those two screens above.
- Offline mode overlays this same store via `offlineDB.js`/`syncQueue.js` (`cacheEntities`, `getCachedEntities`, `saveOffline*`), meaning the integration matrix above is duplicated in an offline variant for every ✅-connected row; the offline variants were **not independently re-traced** in this document and should be a dedicated audit pass if not already covered by `frontend/e2e/offline.spec.js`.
