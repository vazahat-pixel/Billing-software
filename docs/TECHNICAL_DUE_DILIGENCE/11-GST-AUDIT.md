# 11 — GST Compliance Audit

Scope: `backend/services/gstService.js`, `backend/controllers/gstController.js`, `backend/routes/gstRoutes.js`, `backend/models/Sales.js`, `Purchase.js`, `ReturnInvoice.js`, `DebitCreditNote.js`, `Company.js`, `CompanySettings.js`; `frontend/src/pages/gst/GstModals.jsx`, `GSTPage.jsx`, `frontend/src/pages/accounting/GSTReport.jsx`, `frontend/src/utils/gstExport.js`, `frontend/src/utils/invoiceHelpers.js`, `frontend/src/pages/sales/SalesModal.jsx`, `frontend/src/components/InvoicePDFViewer.jsx`. This is the compliance-facing surface a real Chartered Accountant or GST auditor would test first — and the surface most likely to fail a real filing.

> **Headline finding, stated up front:** every printed Tax Invoice PDF generated by this system — for *every* company, *every* customer, using the real production build — displays a hardcoded fictitious seller identity (name **"MAHAVEER TEXTILES PVT. LTD."**, GSTIN **`24AABCM1234D1Z5`**) instead of the logged-in company's actual registered name/GSTIN/address. See section 6. ## 1. Interstate/Intrastate determination — manual selection, not GSTIN comparison

The single most consequential GST-correctness fact about this system: **whether a transaction is taxed CGST+SGST or IGST is a human's manual dropdown choice on the invoice form, not a computed comparison of the two parties' GSTIN state codes.**

```76:80:backend/models/Sales.js
gstType: {
  type: String,
  enum: ['CGST+SGST', 'IGST'],
  default: 'CGST+SGST'
},
```

`SalesModal.jsx` renders `gstType` as a plain `<select>` the operator picks (defaulting to `CGST+SGST`) — it is never derived from comparing `customerId.gstin.substring(0,2)` to the company's own state code. `Purchase.js` has the identical pattern for supplier-side GST. Once the operator picks a `gstType`, the frontend computes `cgst`/`sgst`/`igst` amounts client-side according to that choice and submits them as trusted numbers (per `07-API-AUDIT.md` API-8, the backend never recomputes or validates these against the party's actual GSTIN).

**Downstream effect:** `accountingService.onSalesInvoicePost` / `onPurchaseBillPost` (`accountingService.js:109-178`) simply post whatever `cgst`/`sgst`/`igst` values already sit on the `Sales`/`Purchase` document — there is no server-side inter-state check anywhere in this path. **This means an operator can, by mistake or intentionally, select `IGST` for an intra-state sale (or vice-versa) and the system will accept, post, and report it as filed with zero validation, with no warning surfaced anywhere in the UI or API.**

The **one and only place in the entire codebase with a real, GSTIN-based inter-state determination** is Job Work Charges posting:

```251:267:backend/services/accountingService.js
// Determine GST Type (CGST+SGST or IGST)
const millParty = await Party.findById(receive.millId);
const CompanySettings = require('../models/CompanySettings');
const companySettings = await CompanySettings.findOne({ companyId });

let isInterState = false;
if (millParty && millParty.gstin && companySettings && companySettings.gstin) {
  const millStateCode = millParty.gstin.substring(0, 2);
  const companyStateCode = companySettings.gstin.substring(0, 2);
  if (millStateCode !== companyStateCode) {
    isInterState = true;
  }
} else if (millParty && millParty.state && companySettings && companySettings.state) {
  if (millParty.state.toLowerCase() !== companySettings.state.toLowerCase()) {
    isInterState = true;
  }
}
```
This is the correct algorithm (GSTIN state-code comparison, with a state-name fallback for unregistered parties) — but it exists **only** for the Job Work Charges auto-posting path, covering perhaps the smallest fraction of a textile trading company's total GST-bearing transactions. Sales, Purchases, Returns, and Notes — the overwhelming bulk of transaction volume — all rely on manual/trusted selection instead. This inconsistency (one correct algorithm sitting unused next to four manual-trust paths that need it far more) strongly suggests the Job Work fix was a targeted patch rather than a systemic one.

## 2. GSTR-1 (`GET /api/gst/gstr1`) — structural review

`gstService.getGstr1()` (`gstService.js:8-128`) builds the standard B2B/B2CL/B2CS/HSN sections.

### 2.1 Correct/adequate
- B2B vs B2C split by `gstin.length >= 15` (correct GSTIN length check).
- B2CL vs B2CS split by the ₹2.5 lakh threshold (correct threshold, though see 2.2 for a subtlety).
- HSN summary rate now derived from `line.itemId?.gstRate` per-item (comment marks this `// FIXED: Use actual item GST rate instead of hardcoded 5%` — a prior bug was fixed here), and CGST/SGST split at the item level uses the real per-item rate.
- Place-of-supply (`pos`) resolves to the company's own state (`company?.meta?.state`) — correct for the common case of intra-state or where POS = seller's state, though genuinely POS should be the buyer's state code for most B2B lines, which this does not attempt to derive from the customer's GSTIN.

### 2.2 Defects

**`b2cs.rt` is hardcoded to `5` regardless of the actual item's GST rate:**
```60:69:backend/services/gstService.js
b2cs.push({
  typ: 'OE',
  pos: row.pos,
  rt: 5, // Default GST rate for b2cs summary — will vary per item below
  txval: parseFloat(taxable.toFixed(2)),
  ...
```
The code comment itself concedes the rate "will vary per item below" — but the `rt: 5` value that actually ends up in the exported B2CS row is never corrected to reflect that variance; it stays `5` even when the invoice's real items are taxed at 12%, 18%, or any other GST slab. **Every B2CS row in the exported GSTR-1 JSON misreports its tax rate unless the underlying sale happens to be taxed at exactly 5%** (which, per `09-TEXTILE-DOMAIN.md`, is at least the *common* case for grey/cotton fabric — but wrong for value-added/blended/synthetic textiles at 12%, or accessories/trims at 18%, all of which this business plausibly deals in).

**Default HSN code `5208` is silently substituted whenever `Item.hsnCode` is unset, with no warning surfaced to the filer at export time:**
```73:73:backend/services/gstService.js
const hsn = line.itemId?.hsnCode || '5208';
```
`5208` is *"Woven fabrics of cotton, containing ≥85% cotton by weight, weighing ≤200 g/m²"* — a real, specific HSN code for a real category of cotton fabric. Substituting it as a blanket default for **any** item missing an HSN code (accessories, trims, dyes, chemicals, yarn products taxed under entirely different chapters, non-cotton fabrics) silently misclassifies those goods in the GSTR-1 HSN summary. Because the default is a *plausible-looking real code* rather than an obviously-wrong placeholder like `0000`, a filer skimming the JSON export is unlikely to notice the substitution — this is a worse failure mode than an obvious error would be, since it fails silently and produces GSTR-9 (annual return) HSN-wise summary mismatches that surface only much later, at annual reconciliation time.

**`hash: 'hash'`:**
```110:110:backend/services/gstService.js
hash: 'hash',
```
The exported GSTR-1 JSON schema includes a top-level `hash` field — the real GST portal schema uses this as an internal reference hash for the filed return. Here it is a literal string constant `'hash'`. This is unambiguous placeholder/scaffold code that was never finished; if this JSON were ever fed directly to a GST-suite tool or the portal's offline utility expecting a real hash, it would either be ignored (benign) or cause a validation failure (if the consuming tool checks the field's format) — either way it signals this export was never actually tested against a real filing pipeline.

**`version: 'GST3.2.2'`** is hardcoded rather than tracking the portal's actual current schema version — a cosmetic but telling detail (the offline-tool schema version has moved multiple times since; a hardcoded value means this was written against whatever was current at author-time and never revisited).

**B2CL threshold checked against `taxable`, not `netAmount`/invoice value:** GST law's ₹2.5 lakh (previously ₹1 lakh, revised) B2CL threshold is based on invoice value, and the code does compare against `taxable` (pre-tax) rather than the GST-inclusive invoice total — for high-value low-rate-slab invoices near the boundary, this could misclassify a B2CL invoice as B2CS or vice-versa depending on which figure the real portal actually expects (the GST rules define the threshold in terms of "invoice value," commonly interpreted as `taxable + tax`, not `taxable` alone).

**Returns are entirely absent from the GSTR-1 export.** `getGstr1()` queries **only** the `Sales` collection — `ReturnInvoice` documents (Sales Returns, which under real GST law must be reported as Credit/Debit Notes — Table 9B "Credit/Debit Notes (Registered)" or Table 9B unregistered — in the GSTR-1 filing for the period the return was issued) are never read by this function at all. **Every sales return processed through `POST /api/returns` is invisible to the GSTR-1 export.** A company using this system's GSTR-1 JSON as the actual basis for a real government filing would systematically over-report outward taxable supply by the full amount of every unrecorded sales return, every single period. This is a direct filing-accuracy defect, not a cosmetic one.

**Debit/Credit Notes are also absent from GSTR-1** — same root cause: `getGstr1()` never touches the `DebitCreditNote` collection, and even if it did, per section 4 below the notes carry no GST breakdown to report.

### 2.3 GSTR-1 sub-schema shape review
The exported object's top-level shape (`gstin, fp, version, hash, b2b, b2cl, b2cs, hsn`) roughly mirrors the real GST offline-tool JSON schema's outermost keys, but is missing several mandatory-when-applicable sections entirely: no `cdnr` (Credit/Debit Notes — Registered), `cdnur` (Credit/Debit Notes — Unregistered), `exp` (Exports), `nil` (Nil-rated/Exempt/Non-GST supplies), `at`/`atadj` (Advances), or `doc_issue` (Document summary — mandatory in every GSTR-1, tracking invoice/CN/DN/challan number ranges issued and cancelled). For a textile trading business that issues job-work delivery challans and possibly deals in exempt categories, several of these sections are not just nice-to-have but legally required whenever applicable — their complete absence means this export cannot be a drop-in replacement for a real GSTR-1 filing even before accounting for the bugs above; it is at best a B2B/B2CL/B2CS/HSN-only partial extract.

## 3. GSTR-2 (`GET /api/gst/gstr2`) — flat, not structured

```146:158:backend/services/gstService.js
return purchases.map(p => ({
  invoiceNo: p.invoiceNo,
  date: p.date,
  partyName: p.supplierId?.name,
  gstin: p.supplierId?.gstin,
  taxable: p.taxableAmount || 0,
  cgst: p.cgst || (p.gstAmount || 0) / 2,
  sgst: p.sgst || (p.gstAmount || 0) / 2,
  igst: p.igst || 0,
  gstAmount: p.gstAmount || 0,
  netAmount: p.netAmount || p.taxableAmount
}));
```
Unlike `getGstr1()` (which at least buckets into B2B/B2CL/B2CS/HSN sections mirroring the real portal schema), `getGstr2()` returns a **flat, undifferentiated array of purchase rows** with no B2B/B2BUR/CDNR/ISD/IMPG structure at all — it is not shaped like any real GSTR-2/2A/2B schema, it is simply a re-projection of the `Purchase` collection with renamed fields. In practice, GSTR-2 (supplier-side return) was suspended by the GST Council years ago in favor of auto-drafted GSTR-2A/2B — so building a true "GSTR-2" filing UI is arguably chasing a non-existent target; what a real business actually needs here is a **GSTR-2B reconciliation** (matching this system's purchase register against the government portal's auto-populated ITC statement) — which is precisely the feature `Gst2bMatchingModal` in the frontend *pretends* to provide (see section 5) but does not actually implement. The backend's flat `getGstr2()` has no relationship to that frontend modal at all — the modal doesn't even call this endpoint (confirmed: `Gst2bMatchingModal` reads `purchases` directly from the Zustand store, never calls `fetchGstr2` or hits `/api/gst/gstr2`).

## 4. GSTR-3B — summary only, and only on the CA Dashboard

There is **no dedicated `/api/gst/gstr3b` endpoint at all.** GSTR-3B figures exist only as the `gstr3b` block nested inside `getCADashboard()`'s response (`gstService.js:258-279`):

```258:279:backend/services/gstService.js
gstr3b: {
  outward: { taxable, cgst, sgst, igst, total },
  itc: { taxable, cgst, sgst, igst, total },
  net: { cgst, sgst, igst, total }
}
```
This computes Section 3.1 (Outward Supplies) and Section 4 (Eligible ITC) net figures — but:
- **No Section 3.1(d)** (inward supplies liable to reverse charge) — reverse-charge purchases (e.g., GTA freight, unregistered-dealer purchases where RCM applies) are not distinguished from normal ITC anywhere in the data model (`Purchase` has no `reverseCharge` flag).
- **No Section 3.2** (inter-state supplies to unregistered persons/composition dealers/UIN holders — a breakdown GSTR-3B requires separately from the main 3.1 figures).
- **No Section 5** (exempt/nil-rated/non-GST inward supplies).
- **No Section 6.1** (actual tax payment breakdown — how much was paid via ITC vs. cash, cash ledger vs. credit ledger split) — the `net` block here is simply `outward - itc`, a liability calculation, not a payment record; there is no linkage to any concept of "amount actually paid to the government" anywhere in the system (no GST Payable/Paid voucher type, no GST PLA/Cash Ledger equivalent ledger in `SYSTEM_LEDGER_TEMPLATES`).
- **Returns are excluded** from `outwardTaxable`/`outwardCgst`/etc. for the same reason as GSTR-1 (section 2.2) — `getCADashboard()`'s outward totals are built from `gstr1.invoices`, which never included returns to begin with. The dashboard **does** separately surface a `returnGst`/`noteGst` figure in its `summary` block (see section 4.1) alongside the (return-exclusive) `gstr3b` block, but nothing subtracts `returnGst` from the GSTR-3B net liability figure — a filer would have to manually net these two disjoint numbers themselves, and the UI (`Gst3bMonthlyModal` — see 5.1) doesn't even display `returnGst`/`noteGst` at all, only recomputing its own client-side numbers from `sales`/`purchases` with the same omission.

### 4.1 `noteGst` is always zero — a real, confirmed bug
```231:232:backend/services/gstService.js
const returnGst = returns.reduce((s, r) => s + (r.gstAmount || 0), 0);
const noteGst = notes.reduce((s, n) => s + (n.gstAmount || 0), 0);
```
`DebitCreditNote` (`backend/models/DebitCreditNote.js`) has **no `gstAmount` field defined on its schema at all** — confirmed by direct inspection; the schema's fields are limited to `noteType`, `noteNo`, `date`, `partyLedgerId`, `amount`, `againstInvoiceNo`, and a handful of narration/reference fields, none named `gstAmount`. `n.gstAmount` therefore evaluates to `undefined` for every single note document that has ever existed or ever will, `(n.gstAmount || 0)` always resolves to `0`, and **`noteGst` is unconditionally `0` in every CA Dashboard response, for every company, regardless of how many GST-bearing Debit/Credit Notes have been issued.** This is a direct consequence of the accounting-layer defect documented in `10-ACCOUNTING-AUDIT.md` §3.3 (notes never capture a GST component to begin with) surfacing here as a dead, always-zero reporting field — the two defects should be fixed together, since fixing the GST reporting field without first adding `gstAmount`/`cgst`/`sgst`/`igst` fields to the `DebitCreditNote` schema (and having `noteController.js` actually populate and post them) would have nothing real to read.

## 5. Frontend GST modals — a mix of real and simulated functionality

`GstModals.jsx` contains six modal components. Their real-vs-fake status differs sharply and is not obvious from the UI, which presents all six with equally polished "Compliance Intelligence" / "Audit Engine" styling:

| Modal | Real or Simulated | Evidence |
|---|---|---|
| **`Gst3bMonthlyModal`** | ⚠️ **Real computation, fake filing action.** Client-side recomputes outward/inward CGST/SGST/IGST from `sales`/`purchases` in the store (own bespoke interstate check via `!s.gstin.startsWith('24')` — hardcoded to Gujarat's `24` state code, meaning this check is **wrong for any company registered outside Gujarat**, an even narrower bug than the backend's manual-`gstType` issue). The "Execute Filing" button (`handleFileReturn`) is **entirely fake**: `setTimeout(() => { setFiledStatus(true); }, 2000)` — a 2-second spinner theatrically simulating a government-portal filing call, with no network request, no backend endpoint invoked, and no persisted "filed" state (refreshing the page/reopening the modal loses the `filedStatus` entirely, since it's local `useState`, not written to any `Book`/audit record). |
| **`Gstr1Modal`** | ✅ **Mostly real.** Own client-side B2B/B2CL/B2CS/HSN computation (again with the same hardcoded `!s.gstin.startsWith('24')` inter-state bug), and the "Download JSON Schema" button genuinely calls `fetchGstr1()` → `GET /api/gst/gstr1` and downloads the real backend response via `downloadJson()`. This is a legitimately functional export path, inheriting all of section 2's backend defects but not adding fake UI theater on top. |
| **`Gst2bMatchingModal`** | ❌ **Entirely fake — confirmed simulation, not a stub.** |
| **`Gst3bDetailModal`** | ⚠️ **Real data, but redundant/unlabeled as non-authoritative.** Simple pass-through table of `sales`/`purchases` rows and their stored `totals.cgst/sgst/igst` — no computation bug here since it just displays stored numbers, but it duplicates `Gst3bMonthlyModal`'s data with no cross-reference, and is yet a **third** distinct GST total surface (alongside the backend's two — GSTR-1's inline totals and the CA Dashboard's `gstr3b` block) that could disagree with either. |
| **`Gstr1ErrorChekModal`** | ❌ **Entirely fake — confirmed simulation, not a stub.** |
| **`GstComplianceModal`** | ❌ **Entirely fake, and worse: fabricated to look reassuring.** |


There is no GSTN portal API integration anywhere in this codebase (no OAuth/API-key config for the GST System APIs, no reference to any GSP — GST Suvidha Provider). This modal:
- Sets `portalTaxable = erpTaxable` unconditionally — the "portal value" column is a **literal copy of the local ERP value**, not an independently-fetched figure, so a genuine mismatch could never be displayed even by accident.
- Determines MATCHED/MISMATCH status by **`index % 3 === 0`** — a purely positional modulo check against the array index, with zero relationship to any actual data comparison. The 1st, 4th, 7th... purchase in whatever order the store happens to return them will always show "MATCHED"; the rest always show "MISMATCH" — regardless of which supplier, invoice, or amount is involved.
- Falls back to the literal placeholder string `'24SPLY1234F1Z5'` as a fake GSTIN whenever a supplier's real GSTIN is missing, presenting a fabricated-looking-real value in a field an auditor would reasonably expect to be either the true GSTIN or clearly blank.

### 5.2 `Gstr1ErrorChekModal` — confirmed fake
`runAudit()`'s "scan" is a 2-second `setTimeout` that unconditionally populates **two hardcoded issue objects** referencing fictitious invoices `SI-204` (party "Suresh Fabrics") and `SI-209` (party "Om Textiles") with fixed error codes `GSTIN_INVALID`/`HSN_MISSING` — these invoice numbers and party names do not exist in and are never read from the actual `sales` data fetched by the component (the `fetchSales()` call's result is never even referenced inside `runAudit`). `handleAutoFix()` is equally fake: another 2-second `setTimeout` that simply clears the hardcoded issue list to give the appearance of remediation, with **no backend call, no actual data mutation of any invoice, GSTIN, or HSN field.** Running this "Auto-Fix" against a company's real, genuinely-invalid data (e.g. a real malformed GSTIN on a real invoice) would report success while changing nothing.

### 5.3 `GstComplianceModal` — confirmed fake, and the most misleading of the six
This modal has **no data-fetching logic at all** — no `useStore()` call, no `useEffect`, no props. Every number is a literal JSX constant: `98.2` aggregate rating, `₹ 14.2K` ITC Matching Gap with a fabricated `2.1% Unmatched` badge, `100%` Liability Coverage labeled "Fully Reconciled", `2h` Portal Sync Age labeled "Real-time", and a "Compliance Checklist — FY 2026-27" grid where APRIL/MAY/JUNE/JULY all show green checkmarks for both GSTR-1 and GSTR-3B **unconditionally, for every company, regardless of whether that company has filed anything, has any sales at all, or even exists yet.** A brand-new company created five minutes ago with zero transactions would see an identical "98.2 / A+ / 100% Fully Reconciled / all months filed" scorecard as a ten-year-old company with a genuinely clean filing history. This is not an incomplete feature — it is a component whose entire purpose is to display a fabricated positive compliance signal, which is the single most reputationally dangerous piece of fake functionality identified across all six audit documents: a business owner or investor glancing at this screen would reasonably conclude their GST compliance is verified and excellent when in fact **no compliance verification of any kind has occurred.**

## 6. Invoice PDF uses a hardcoded fictitious company identity — critical finding

```1:16:frontend/src/utils/invoiceHelpers.js
  phone: '+91 98765 43210',
```

`InvoicePDFViewer.jsx` accepts `company` as a prop **defaulting to `DEMO_COMPANY`** (`company = DEMO_COMPANY` in its destructured props). Its two production callers — `SalesPrint.jsx` and `PurchasePrint.jsx` — **never pass a `company` prop at all**:

```13:21:frontend/src/pages/sales/SalesPrint.jsx

  <InvoicePDFViewer
    type="sale"
    invoice={invoice}
    parties={parties}
    items={items}

```
No `company={...}` prop anywhere in the JSX. This means **every rendered/downloaded/WhatsApp-shared invoice PDF in the entire product, for every company account, past and future, displays "MAHAVEER TEXTILES PVT. LTD.", GSTIN `24AABCM1234D1Z5`, PAN `AABCM1234D`, and a fictitious HDFC Bank account** as the *seller* — regardless of which real company is actually logged in, what their real registered name is, what their real GSTIN is, or what bank account they actually operate. The `buildWhatsAppMessage()` helper used for the "Share via WhatsApp" feature has the identical default and is called with the same un-overridden `company` prop, so the WhatsApp-shared invoice summary sent to a real customer also cites the fake company name.

**Why this matters for GST specifically:**
- Rule 46 of the CGST Rules mandates that a tax invoice show the supplier's **name, address, and GSTIN** — the invoice being generated shows someone else's (fictitious) GSTIN entirely. A recipient attempting to claim ITC based on this PDF, or a tax officer inspecting it, would find a GSTIN that identifies a different (non-existent) entity than the one that actually supplied the goods.
- The company's *own* real GSTIN (`Company.meta.gstin` / `CompanySettings.gstin`) exists in the database and is used correctly by the **backend** GST reports (`gstService.getGstr1()` reads `company?.meta?.gstin`) — the defect is isolated entirely to the **PDF-rendering/print layer**, which never wires the already-available real company data through to `InvoicePDFViewer`. This is very likely leftover demo/scaffolding code (the file's own comment literally says *"replace with live config later"*) that was never revisited before the print feature was considered "done."
- This is trivially fixable (thread the real `currentCompany`/`CompanySettings` data as the `company` prop at both call sites) but, as shipped, is a **launch-blocking legal/compliance defect** that supersedes almost every other finding in this document in severity — a business cannot legally issue invoices under a name and GSTIN that are not its own.

## 7. Miscellaneous GST-adjacent facts

- **`Item.hsnCode`** has no format validation (no regex/length check ensuring 4, 6, or 8 digits per the applicable turnover-based HSN digit requirement) — any string is accepted at the master-data level, meaning the `5208` default substitution problem (section 2.2) is compounded by upstream masters potentially containing malformed HSN codes to begin with, with no validation catching either.
- **`Item.gstRate`** is a free numeric field with no enum constraint to the actual legal GST slabs (0%, 0.25%, 3%, 5%, 12%, 18%, 28%) — an operator can type `17` or `5.5` and the system will accept and post it as-is, computing GST at a rate that does not exist under any GST notification.
- **No e-Invoice (IRN/QR code) generation or integration exists anywhere** — for businesses crossing the e-invoicing turnover threshold (₹5 crore aggregate turnover, per current CBIC notifications), a valid tax invoice must carry an IRN and QR code from the government's IRP; this system has no IRP integration, no IRN field on `Sales`, and no QR code rendering on `InvoicePDFViewer`.
- **No e-Way Bill generation or threshold tracking** — consignment-value-based e-way bill requirements (relevant for job-work goods movement between mills/dyers, a core textile workflow per `09-TEXTILE-DOMAIN.md`) are entirely unaddressed; `Job` schema has no delivery-challan or e-way-bill-number field.
- **`Company.meta.state`** (used as the GSTR-1 "place of supply" fallback in section 2.1) and `CompanySettings.state`/`.gstin` (used by the one real inter-state check in section 1) are **two separate, independently-editable fields on two separate models** representing what should be the same fact about the company — nothing keeps them in sync, so a company whose `CompanySettings.state` was updated but whose `Company.meta.state` was not (or vice versa) will get self-contradictory GST behavior depending on which code path executes.

| GSTR-2B reconciliation (the feature businesses actually need) | ❌ Not implemented — the UI that claims to do this (`Gst2bMatchingModal`) is fully simulated. |
| GSTR-3B | ⚠️ Net liability summary only, on CA Dashboard only, no dedicated endpoint, missing RCM/exempt/payment-ledger sections, returns/notes exclusion inherited from GSTR-1. |
| Debit/Credit Note GST handling | ❌ Broken at the schema level — no GST fields exist, `noteGst` is permanently `0`. |
| Frontend GST "intelligence" modals | ❌ 3 of 6 modals (`Gst2bMatchingModal`, `Gstr1ErrorChekModal`, `GstComplianceModal`) are confirmed fully-simulated UI theater with hardcoded/fabricated results and zero backend integration. |
| Invoice PDF legal validity | ❌ **Critical** — every invoice shows a fictitious seller identity/GSTIN, unrelated to the actual company. |
| e-Invoicing / e-Way Bill | ❌ Not implemented at all. |

**Verdict:** the backend GST reporting service (`gstService.js`) is a genuine, mostly-competent piece of engineering for the slice of GST it covers (GSTR-1 B2B/B2CL/B2CS/HSN, a GSTR-3B-shaped summary) — it is let down by real, fixable bugs (hardcoded B2CS rate, wrong HSN default, returns/notes exclusion) rather than being fundamentally unsound. The frontend GST module, by contrast, mixes this real backend data with three modals that are **outright fabricated demonstrations with no live computation at all**, presented with identical visual polish to the real ones — a serious risk for any user or investor evaluating the product's actual compliance-readiness at face value. The invoice-PDF company-identity bug is the single highest-severity finding in this file and should be treated as launch-blocking ahead of every other item.
